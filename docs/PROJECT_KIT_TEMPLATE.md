# DiffshipOS Project Kit (template)

This document is a **template** used by `diffship init`.

Goal: help an AI agent reliably produce **diffship-compatible patch bundles** and follow the safety + workflow contracts.

---

## 1. What diffship is

diffship is a local tool that:

- applies AI-returned **patch bundles** safely (isolated sandbox)
- records deterministic run logs under `.diffship/runs/`
- can generate “reprompt bundles” for iteration (`pack-fix`)
- (planned/spec) will also support building AI handoff bundles from local diffs

diffship does **not**:

- run arbitrary commands generated by the AI
- push to remotes / open PRs

---

## 2. Golden rules (non-negotiables)

1) **Base commit must match**
- Patch bundles MUST include `manifest.yaml` with `base_commit`.
- The patch MUST be generated against that exact commit.

2) **Paths are strict**
- Repo-relative only.
- No absolute paths, no `..` traversal.
- Never touch `.git/` or `.diffship/`.

3) **No secrets**
- Never include real tokens/keys.
- If environment values are needed, provide templates and user tasks.

4) **Determinism**
- File ordering and outputs should be stable.

---

## 3. Patch Bundle contract (AI → diffship)

### 3.1 Directory structure

Create a zip that contains exactly one top-level directory:

```text
patchship_YYYY-MM-DD_HHMM/
  manifest.yaml
  changes/
    0001.patch
  summary.md              # optional
  constraints.yaml        # optional
  checks_request.yaml     # optional
  commit_message.txt      # optional
  tasks/                  # optional
    USER_TASKS.md
    TASKS.yaml
    ENV_TEMPLATE.env
```

### 3.2 `manifest.yaml` required fields

Required:
- `protocol_version`: e.g. `1`
- `created_at`: ISO8601
- `task_id`: string
- `base_commit`: full git SHA
- `apply_mode`: `git-apply` or `git-am`
- `touched_files`: list of repo-relative paths

Recommended:
- `requires_docs_update`: bool
- `requires_plan_update`: bool
- `tasks_required`: bool
- `secrets_ack_required`: bool

### 3.3 Patch limitations (MVP)

Unless the repo explicitly supports it, avoid these features:
- `GIT binary patch`
- submodule changes
- file mode changes
- rename/copy metadata

---

## 4. Commit behavior (diffship policy)

Patch format (`apply_mode`) is independent from whether diffship commits.

- The AI may include `commit_message.txt`.
- Whether diffship auto-commits is decided by user config:
  - `manual`: never commit automatically
  - `auto`: commit after a successful apply + verify

When `commit_message.txt` is missing, diffship may use a deterministic fallback template.

---

## 5. User tasks (env, migrations, manual steps)

If the user must do something manually (e.g., create `.env`, rotate tokens, run a one-off migration), include `tasks/USER_TASKS.md`.

Guidelines:
- Make tasks **checklist-style**.
- If env vars are needed, include `tasks/ENV_TEMPLATE.env` with placeholder values.
- Never include real secrets.

Example `tasks/USER_TASKS.md`:

```md
## Required user tasks

- [ ] Create `.env` from `tasks/ENV_TEMPLATE.env`
- [ ] Set `API_KEY` (do not commit)
- [ ] Re-run `diffship verify --profile standard`
```

---

## 6. How to iterate with diffship

Typical loop:

1) AI produces a patch bundle zip
2) User runs:

```bash
diffship loop path/to/patch-bundle.zip
```

3) If it fails, the user runs `diffship pack-fix --run-id <run-id>` to generate a reprompt zip under `.diffship/runs/<run-id>/...`
4) User sends that reprompt bundle back to the AI
5) AI fixes and returns a new patch bundle with an updated `base_commit` (if the session advanced)

---

## 7. What to include in the AI response

When returning a patch bundle, include:

- the zip itself
- a short explanation of intent
- any critical warnings (e.g., tasks required)

Avoid:
- long prose
- dumping full file contents beyond what’s needed
